- `desmos-proto::antireplay::AntiReplayWindow` 128-bit sliding anti-replay bitmap, shape modelled after RFC 6479 §3. Tracks the highest accepted sequence in `u64` plus a `u128` bitmap where bit `k` represents `highest - k`; `check_and_update(seq)` accepts in-order and out-of-order within the 128-wide window, rejects duplicates and out-of-window via a typed `ReplayError { Duplicate, OutOfWindow }`. `would_accept(seq)` is the non-mutating probe for speculative pipeline stages; `highest()` and the `new` constructor round out the API. The first packet on a fresh window is always accepted (even `seq == 0`) thanks to an `initialised` flag that separates "never seen anything" from "zero is the current highest". 12 unit tests cover every branch (fresh init, in-order bursts, out-of-order within window, exact window edge at offset 127, one-past-edge rejection, duplicate at highest and within window, WINDOW_SIZE forward jump overflow branch, large forward jump clearing the window, `would_accept` agreement, and clone independence). 4 integration tests in `crates/desmos-proto/tests/antireplay.rs` verify the real window against a HashSet-based `Reference` model: in-order, reverse-order, a pathological manually-crafted stream, and a deterministic 10 000-stream xorshift64 fuzzer where each stream runs 64 packets biased across the five branches (near cursor, recent duplicate, big forward jump, ancient out-of-window, exact cursor duplicate) and every real decision must match the reference exactly — stronger than the Task 18 "no false accepts" acceptance bar.
- `desmos-core::pipeline::{forward_tun_to_udp_encrypted, forward_udp_to_tun_encrypted}` Phase 2 encrypted packet pipeline stages sitting alongside the Phase 1 plaintext path. Outbound reads an IP packet from the TUN, calls `Session<Established>::encrypt_packet` (new API returning `(seq, ciphertext)`), stamps the assigned sequence into the DWP header's `Seq` field (narrowed from u64 to u32; sessions rekey well before the upper half is relevant), and sends `[16-byte DWP header] [ciphertext + 16-byte Poly1305 tag]` to the peer. Inbound recvs the datagram, decodes the header, widens the 32-bit wire sequence back to u64, calls `Session<Established>::decrypt_data` which enforces the per-session anti-replay window before the AEAD, and delivers the plaintext to the TUN. Drops are silent on the wire but each error class increments a specific counter: `bad_header` for malformed DWP headers, `replay_drops` for anti-replay rejections, `decrypt_failures` for tag mismatches / wrong-session / tampered ciphertext.
- `desmos-core::pipeline::PipelineMetrics` atomic counter block (`packets_sent`, `packets_received`, `bytes_sent`, `bytes_received`, `bad_header`, `decrypt_failures`, `replay_drops`) plus `record_*` helpers and a `MetricsSnapshot` struct for lock-free reads. 4 unit tests cover zero-init, sent/received recording, and independent error counters.

- `desmos-core::net` host interface discovery and Linux link-state watching. `net::iface::list()` returns `Vec<NetworkInterface>` with name, 6-byte MAC, `Vec<Ipv4Addr>`, `Vec<Ipv6Addr>`, an `IfaceFlags` bitfield decoded from `/sys/class/net/<name>/flags`, and the `operstate` string. On each burst the loop fires the ping at `peer_primary` and at every `peer_alt_ports[n]` candidate (best-effort symmetric-NAT fallback — callers generate the list from `port ± N` or a birthday-attack span), then blocks on `recv_from` for up to `interval_ms`. Incoming `Ping(nonce)` from anyone is echoed back as `Pong(nonce)` so the *peer* side's own hole-punch call can finish on our reply path — both sides can complete even when neither's initial burst lands first. `Pong` with a mismatched nonce is silently dropped; only the matching nonce returns the confirmed `SocketAddr` as the address the bonding engine installs for this link. `P2pError` taxonomy: `Timeout / BadConfig(&'static str) / Io(String)`. 13 unit tests cover encode / decode round-trip for both kinds, wrong-length and unknown-magic rejection, nonce uniqueness across calls, the zero-deadline / zero-interval `BadConfig` branches, a loopback cone-NAT happy path through a fake peer thread, an explicit assertion that the caller's loop echoes a peer-initiated `Ping` back as a matching `Pong`, the symmetric-NAT fallback finding the peer on an `alt_port` when the primary port is a dead black hole, the stray-`Pong`-wrong-nonce path (retries until the deadline), the silent-peer timeout, and `Display` coverage of every variant. - `desmos-rt::windows::reactor::IocpReactor` Windows IOCP reactor behind the `Reactor` trait. Hand-declared `extern "system"` FFI bindings for `CreateIoCompletionPort`, `GetQueuedCompletionStatusEx`, `CloseHandle`, `WSARecv`, `WSASend`, `WSAGetLastError` — no `windows-sys` or `winapi` crate. Bridges the completion-based IOCP model to the readiness-based `Reactor` trait via the **zero-byte overlapped recv/send** technique: `register` associates the socket with the IOCP and submits zero-byte `WSARecv` / `WSASend` probes; when the kernel completes them (data available / send buffer free), `poll` dequeues the completion, maps it to `READABLE` / `WRITABLE`, and re-arms the probe for the next poll cycle. Per-socket state (`SocketState`) holds two `Box<Overlapped>` allocations (one for read, one for write) with `read_pending` / `write_pending` guards to avoid double-arming. `reregister` updates the token and arms newly requested probes; `deregister` removes tracking (in-flight completions are silently discarded on the next poll). `poll` uses `GetQueuedCompletionStatusEx` (batched dequeue), handles `WAIT_TIMEOUT` as zero events, and coalesces events with the same token. - `desmos-core::p2p::relay` relay fallback for P2P tunnels closing Phase 5. When UDP hole punching fails to establish a direct path, both peers fall back to routing traffic through a Desmos server acting as a relay from the `[p2p].relay_servers` config list. Wire format: 16-byte header (12-byte `DESMOS-RELAY` magic + 1-byte command + 1 reserved + 2-byte BE payload length) + variable payload. Five command codes: `Register` (peer → relay, carries 32-byte peer public key), `Registered` (relay → peer, ack), `Data` (bidirectional opaque DWP forwarding), `PeerJoined` (relay → peer, other peer arrived), `Error` (relay → peer, UTF-8 reason). `RelaySession` wraps a confirmed relay endpoint with `send(socket, payload)` / `recv(socket, buf)` for transparent packet forwarding — the relay never inspects the DWP payload (end-to-end encrypted). `relay_connect(socket, relay_addr, peer_key, deadline)` sends `Register`, waits for `Registered` + `PeerJoined` within the deadline. `relay_connect_any(socket, servers, key, deadline)` tries each relay server sequentially with an evenly split time budget (500 ms minimum per attempt), returning the first successful session. `try_direct_then_relay(socket, P2pConnectConfig)` is the top-level orchestrator: runs hole punching first, and on `Timeout` falls back to relay with whatever time remains from `total_deadline_ms`, returning `P2pOutcome::Direct(addr)` or `P2pOutcome::Relayed(session)`. `RelayError` taxonomy: `HolePunch(P2pError) / NoRelay / RelayRejected(String) / BadFrame(&'static str) / NoRelayServers / Io(String) / Timeout`. 20 unit tests cover frame codec round-trip for Register/Data/empty-payload, oversized payload rejection, undersized buffer rejection, short/wrong-magic/unknown-command/truncated-payload decode rejection, exhaustive `RelayCmd::from_byte`, loopback mock-relay connection + data echo round-trip, first-reachable server selection with a dead first entry, empty server list rejection, dead-relay timeout, relay session send+recv data with echo, error frame surfacing, direct-succeeds-skips-relay orchestration, falls-back-to-relay-when-punch-times-out, both-fail error propagation, and `Display` coverage of every `RelayError` variant.
